# UnoDOS 8088 port — genuine IBM PC/XT hardware fidelity

## What this port is

Unlike the 68K / 6502 / 65816 / MIPS ports (which re-express UnoDOS on a
*different* CPU), the x86 reference build already **is** the 8088 target. The
"8088 port" is therefore not a rewrite — it is the **real-hardware-fidelity
milestone** for the native build: proving (and fixing) UnoDOS on a genuine
Intel 8088 / IBM PC-XT, following the house method *scripted/real-emulator →
real hardware*.

### Why it was never actually validated

The 2026-06 audit found the kernel + boot chain + apps used 186+/386+
instructions in ~430 places (`PUSHA`/`POPA` → `JO`/`JNO`, `MOVZX` → `POP CS`,
`SHL r,imm` / `PUSH imm` → `RET imm16` / `JS` on real silicon). That pass is
**done and assembler-verified**: everything on the floppy boot path assembles
under `cpu 8086`, with the FAT16/IDE driver bracketed `cpu 386` and runtime-
gated. But it was only ever *run* on QEMU (`qemu-system-i386 -M isapc`), which
emulates a 486-class CPU — so none of the genuinely 8088/XT-specific behaviour
was ever exercised:

- 186+/386+ opcode aliasing (invisible on 386+; fatal on 8088)
- the XT keyboard controller (**8255 PPI**, port 0x61 ack) vs. the AT **8042**
- CGA-only reality and CGA snow; the real 4.77 MHz timing envelope
- the real installed-RAM envelope (the README's "128 KB minimum" claim)
- **serial** mouse (a real XT has no PS/2 port) vs. the INT 15h/C2 + KBC paths

## The rig

[MartyPC](https://github.com/dbalsom/martypc) — a cycle-accurate 8088 emulator
validated against real silicon — booting the open **GLaBIOS** (ROM-free, the
house rule). Machine `unodos_xt` = IBM 5160 (XT), 8088 @ 4.77 MHz, 640K, CGA,
1.44M floppies, Microsoft serial mouse on COM1. Capture via MartyPC's own
framebuffer screenshot (clean under RDP). See
[../tools/xt/README.md](../tools/xt/README.md).

## Feature parity — ACHIEVED (2026-06-15, v3.30.0 / Build 410)

The native build runs at full feature parity on a cycle-accurate Intel 8088,
verified by harness screenshots in `tools/xt/shots/`:

| Capability | On the emulated 8088 / IBM PC-XT |
|---|---|
| Boot chain | boot sector → stage2 → 104-sector kernel load (INT 13h) ✓ |
| Desktop + window manager | CGA 4-colour desktop, z-order/drag/close ✓ |
| Cooperative scheduler | "Tasks: 2 running" (launcher + app) ✓ |
| Keyboard | custom INT 09h via the **XT 8255 PPI** path ✓ |
| Mouse | **Microsoft serial mouse on COM1 / IRQ4** — motion (both axes), buttons, double-click launch ✓ |
| Storage (read) | FAT12 mount, dir listing (Files), every app loads from floppy ✓ |
| Storage (write) | FAT12 save (Notepad → "XTTEST", title confirms) ✓ |
| CGA apps | SysInfo, Settings, Files, Paint, Clock, Notepad, Music, Tracker, Pac-Man ✓ |
| Sound | PC-speaker apps (Music/Tracker) run (audio not screenshot-verifiable) ✓ |
| Text rendering | the `draw_char` CGA row-blit fast path (one MUL/row, one RMW/byte) is active in CGA mode ✓ |

**Documented deviations (real envelope, not fake parity):**
- **VGA apps are out-of-envelope on a CGA 5150/5160.** The mode-13h/12h apps
  (Dostris VGA, OutLast VGA, Pac-Man VGA) render incorrectly because the CGA
  card cannot do those modes (`m2_vga_out_of_envelope.png`). The CGA app set is
  the XT roster; the VGA variants need an EGA/VGA card.
- **Minimum RAM is 256 KB** (desktop + one app); 640 KB for the full 5-app
  multitasking envelope. 128 KB boots the kernel but not the launcher.
- **Full-screen game/graphics repaint is slow at 4.77 MHz** (e.g. the Pac-Man
  maze paints visibly row-by-row) — the inherent 8088 envelope, like the Apple
  II's 1 MHz. Text already uses the fast path; the game pixel-fill paths are the
  next optimization target if desired.

**Pending (hardware-blocked, as for every other port):** cross-boot floppy
*persistence* writes through to the host image only via MartyPC's GUI menu, and
real INT 13h write timing + the whole stack on a **physical IBM PC/XT** remain
the final real-hardware pass.

## Milestones

### M0 — cycle-accurate XT rig + boot (DONE, 2026-06-14)

Stood up the MartyPC rig and **booted the project's primary
`build/unodos-144.img` end-to-end on an emulated IBM PC/XT (8088 @ 4.77 MHz,
CGA)** — the first time UnoDOS has run on anything other than a 486-class QEMU.

Verified, harness-captured (`tools/xt/shots/`):

- **Boot chain on real 8088** — boot sector → "Reset disk OK / Load stage2 OK /
  Jump to stage2 / Loading kernel…" → the 104-sector kernel loads. Confirms the
  audit's 8086-clean `boot.asm` + `stage2.asm` fixes hold on real silicon
  (`m0_splash.png`).
- **Full CGA desktop** — the "UnoDOS 3" splash → the 4-colour desktop with the
  15-icon grid, footer "v3.29.0 Build 408" (`m0_desktop.png`).
- **Keyboard works through the XT 8255 PPI** — pressing Enter on the desktop
  launches **SysInfo**, whose window reports "Boot: Floppy", "Tasks: 2 running"
  → the **window manager, app loader, and cooperative scheduler all run on the
  8088** (`m0_sysinfo.png`). This is the headline result: the kernel's INT 0x80
  dispatch, the launcher, and the scheduler — all of which the audit flagged as
  186+/386+-dependent — execute correctly on genuine 8088 silicon.
- **Boots on 256K too** — the desktop comes up on the `unodos_xt_256k` machine
  (`m0_desktop_256k.png`); see the RAM-floor finding below.

### M0 findings (feed M1/M2)

1. **RAM floor — the README's "128 KB minimum" is wrong.** The launcher loads
   at `0x2000:0000` (needs RAM through ~192K just to show the desktop), the heap
   is `0x8000` and the clipboard/dialog scratch is `0x9000` (≈576–640K). The
   desktop + one app (slot 0 at `0x3000`) fit in 256K, but any heap/clipboard/
   file-dialog use reads/writes past installed RAM on a <640K machine. M1 must
   (a) measure the true floor and (b) either gate those features by detected RAM
   or correct the documented minimum. The honest minimum for the full feature
   set is **640K**, not 128K.
2. **Serial mouse not driven.** The cursor renders but is static — UnoDOS's
   mouse paths are INT 15h/C2 (PS/2 BIOS) primary and KBC/IRQ12 fallback, both
   AT-class. A real XT mouse is a **Microsoft serial mouse on COM1**; the rig
   provides one but there is no driver yet (TODO "Serial mouse support"). → M2.
3. **Boot is slow (~30 s to desktop)** at real 4.77 MHz floppy/CGA speed — the
   M3 `draw_char` CGA row-blit fast path (~10× text) is the lever.

### M1 — RAM floor verified; keyboard/WM/app-set exercised (DONE)

The CGA app sweep is complete (see the parity table above): SysInfo, Settings,
Files, Paint, Clock, Notepad, Music, Tracker, Pac-Man all launch and render on
the emulated XT through the keyboard-driven launcher.


- **RAM floor measured (finding 1 resolved).** On the `unodos_xt_128k` machine
  the kernel boots and paints its version/build splash, but the **desktop never
  comes up**: the launcher loads at segment `0x2000` (linear 128–192K), which a
  128K machine does not have, so `auto_load_launcher` fails to `halt_loop`. The
  256K machine boots to the desktop and runs one app; the full 5-app + heap
  (`0x8000`) + clipboard/scratch (`0x9000`) envelope needs 640K. The README's
  "128KB minimum" claim was corrected to **256K (desktop) / 640K (full)** across
  README.md and docs/FEATURES.md.
- **Keyboard + WM** verified in M0 (Enter-launch of SysInfo through the XT 8255
  PPI path). Remaining M1: a deeper sweep of the CGA app set (Files, Notepad,
  Paint, Dostris, Pac-Man) and the keyboard-driven WM (arrows/Tab/ESC) on the
  emulated XT; confirm the keyboard ack is the XT 8255 form, not an AT-only path.

### M2 — XT-appropriate I/O (serial mouse DONE)

- **Microsoft serial mouse on COM1 — DONE & verified** (the headline XT input
  device; finding 2 resolved). New kernel code: `install_serial_mouse` (UART set
  to 1200/7N1, DTR/RTS power-cycle, `'M'` identifier probe, IRQ4 armed) and the
  `int_0C_handler` IRQ4 ISR (3-byte Microsoft framing → cursor position +
  buttons, mirroring the existing PS/2 deferred-cursor + event model). On a
  pre-AT machine `install_mouse` now falls through from `.skip_kbc` to the
  serial probe instead of giving up; `mouse_diag='C'`. Verified end-to-end in
  MartyPC (host mouse captured, relative motion injected): the cursor tracks
  both axes with correct direction, and a **double-click launches Settings**
  (`tools/xt/shots/m2_mouse_*.png`). Build 409.
- **FAT12 write verified** — Notepad saves a file (Ctrl+S → Save dialog →
  "XTTEST"); the title updates from "(untitled)" to "XTTEST", which the editor
  only does on a successful write (`m2_save_done.png`). The full FAT12 read+write
  stack runs on the 8088.
- **VGA apps documented out-of-envelope** — launching Dostris VGA on the CGA
  machine renders incorrectly (`m2_vga_out_of_envelope.png`): the CGA card has no
  mode 13h. The CGA app set is the XT roster; this is a real deviation, like the
  macplus 1-bit gamut.
- Remaining (polish, not parity-blocking): CGA snow avoidance; cross-boot floppy
  persistence (MartyPC writes back only via its GUI menu) + real INT 13h write
  timing belong to the physical-hardware pass.

### M3 — 4.77 MHz performance + real hardware

- **`draw_char` CGA row-blit fast path — already implemented** (`draw_char_cga_fast`):
  one `MUL` per glyph row (not per pixel) and one read-modify-write per VRAM byte,
  active in CGA mode via `dcf_check`. The earlier TODO entry was stale. All XT
  text renders through it.
- The remaining 4.77 MHz cost is full-screen **game/graphics pixel-fill** (e.g.
  the Pac-Man maze repaint) — the inherent 8088 envelope, a documented deviation
  (cf. the Apple II at 1 MHz). A dirty-region/fill fast path is the optional next
  optimization.
- **Physical IBM PC/XT** is the one remaining real-hardware pass — hardware-
  blocked here, exactly as for every other UnoDOS port.
